# External Processor Scenarios

Tests covering external tool integration via the processor pipeline:
stdin handoff, `qed:file()` materialization, alias bypass with `\`,
and processor pipeline composition with external tools.

All external commands are replaced by mock scripts generated by the test harness
from declarations in the manifest. No real external tools are required.

---

## Mock Declaration Format

Each scenario declares a `[[scenario.mock]]` array of mock entries.
The harness generates a stateful mock script for each distinct command name,
places it in a temp `bin/` directory, and prepends that directory to `PATH`
before running the invocation.

**Fields:**

- `command` вЂ” the script name placed on PATH (required)
- `input` вЂ” expected content delivered to the command, either via stdin or via the file
  at `${QED_FILE}`; the harness validates actual content against this file regardless
  of delivery mechanism (optional)
- `expected_args` вЂ” list of expected positional arguments; mock fails if actual args differ (optional)
- `stdout` вЂ” static stdout written on success (optional, defaults to empty)
- `stderr` вЂ” static stderr written on success (optional, defaults to empty)
- `exit_code` вЂ” static exit code returned on success (optional, defaults to 0)

**Sequencing:** multiple declarations for the same `command` form a call queue in
declaration order вЂ” the first declaration handles the first call, the second handles
the second, and so on.
The harness generates a single script per command name that tracks a call counter
in a temp state file. On each invocation it increments the counter and dispatches
to the Nth declaration. If the command is called more times than there are declarations,
the mock exits non-zero with a diagnostic. After the invocation completes, the harness
verifies that every declaration was reached вЂ” any unconsumed declaration is reported
as a harness error. Declaration order must match the order qed actually invokes the
command, which follows source order in the input stream.

**Validation failure:** when any input check fails, the mock exits non-zero and
writes a diagnostic to stderr. This surfaces as a processor error in qed,
causing the scenario to fail because the overall output or exit code will not
match the golden.

## Script Files

### `scripts/external-stdin.qed`

The entire stream is passed to `upcase` via stdin.

```
at() | upcase
```

### `scripts/external-file.qed`

`qed:file()` materializes the region; `${QED_FILE}` is passed as the first positional arg.

```
at() | qed:file() | sortfile "${QED_FILE}"
```

### `scripts/external-args.qed`

`prefixer` is called with `"> "` as its argument; it prepends that string to each line.

```
at() | prefixer "> "
```

### `scripts/alias-bypass.qed`

`upper` is defined as a `qed:upper()` alias.
`\upper` bypasses the alias and resolves `upper` from PATH вЂ” the mock.
Only `bar` is selected; `foo` and `baz` pass through unchanged.

```
upper=qed:upper()
at("bar") | \upper
```

### `scripts/external-pipeline.qed`

Two external tools chained. `upcase` converts to uppercase; `passthrough` passes
the result through unchanged, demonstrating multi-step external pipelines.

```
at() | upcase | passthrough
```

### `scripts/external-replace-pipeline.qed`

`extract-name` is used as the pipeline replacement in `qed:replace()`.
It receives the matched JSON object as stdin and returns just the name field.

```
at(/^\{/) | qed:replace(/(\{.*\})/, extract-name -r ".name")
```

## Notes

### Mock multiplicity in `external-replace-pipeline`

`external-replace-pipeline` declares two entries for `extract-name`, one per
matched line in `json-lines.txt`.
The harness generates a single stateful mock script that tracks a call counter in
a temp state file.
The first call uses the first declaration (alice), the second uses the second (bob).
If qed calls `extract-name` a third time, the mock exits non-zero with a diagnostic.
After the invocation completes, the harness verifies that both declarations were
reached вЂ” if only one was called, the unconsumed declaration is reported as a failure.
The declared order must match the order qed processes matched regions, which follows
source order in the input file.

### `external-file` and `${QED_FILE}` validation

The `sortfile` mock declares both `expected_args` and `input`.
The harness generates a script that:

1. Validates that `$1` equals `${QED_FILE}` (confirming the path was passed as an arg)
2. Compares the declared `input` file against the actual input received вЂ”
   if `${QED_FILE}` is set, reads from that path; otherwise reads from stdin
3. If both checks pass, writes `mocks/stdout/three-lines-sorted.txt` to stdout and exits zero
4. If either check fails, writes a diagnostic to stderr and exits non-zero

The generated mock script handles stdin vs file delivery transparently by checking
whether `${QED_FILE}` is set at runtime. The harness does not need to parse qed
scripts or understand pipeline structure вЂ” delivery mechanism is self-evident to
the mock at execution time.

### `alias-bypass` and PATH

`\` bypasses user-defined alias resolution and resolves the command from PATH.
Since qed processor names always live in the `qed:` namespace, alias names are
always user-space вЂ” `\` is therefore always a choice between an alias and a PATH
command, never between an alias and a qed internal.

The `upper=qed:upper()` alias in the script means unescaped `upper` resolves to
the alias; `\upper` bypasses this and finds the `upper` mock on PATH instead.
The mock returns `BAR` followed by a `[path:upper]` marker line that `qed:upper()`
would never produce, making the two code paths distinguishable in the golden output.
If alias bypass were broken, qed would apply `qed:upper()` internally, producing
plain `BAR` with no marker вЂ” the golden comparison would catch it immediately.
