# Test Harness Specification

Specification for the `qed` integration test harness.

---

## Overview

The harness runs every scenario defined in the `tests/` directories and reports
pass/fail to `cargo test`.
Each scenario is an independent `Trial` registered via `libtest-mimic`.
A scenario consists of a manifest entry, input files, script files, golden files,
and optional mock declarations for external commands.

The harness is split across two layers:

- **Rust layer** — manifest parsing, Trial registration, temp directory
  lifecycle, `scenario.sh` generation, subprocess execution, and pass/fail
  reporting to `cargo test`
- **Bash layer** — file setup, mock script generation, invocation execution,
  golden comparison, and unconsumed mock detection

---

## Workspace Layout

```
tests/
  integration.rs             # libtest-mimic entry point
  harness/
    run-scenario.sh          # main bash runner, invoked once per Trial
    compare-golden.sh        # golden comparison logic
    generate-mock.sh         # mock script generation logic
  selectors/
    manifest.toml
    inputs/
    scripts/
    goldens/
      stdout/
      stderr/
      output/
  external-processors/
    manifest.toml
    inputs/
    scripts/
    mocks/
      input/
      stdout/
      stderr/
    goldens/
      stdout/
      stderr/
      output/
  <other suite directories>/
```

`integration.rs` is the `libtest-mimic` entry point.
At startup it discovers all `manifest.toml` files under `tests/`, parses them,
and registers one `Trial` per scenario.

---

## Rust Layer

### Manifest Parsing

For each `tests/<suite>/manifest.toml` discovered, the Rust layer parses every
`[[scenario]]` entry into a structured value carrying:

- `id` — unique scenario identifier within the suite
- `description` — human-readable description
- `script` — script filename, resolved against `<suite>/scripts/`
- `input` — input filename, resolved against `<suite>/inputs/`
- `stdout` — stdout golden filename, resolved against `<suite>/goldens/stdout/`
- `stderr` — stderr golden filename, resolved against `<suite>/goldens/stderr/`
- `output` — output golden filename, resolved against `<suite>/goldens/output/`
- `exit_code` — expected exit code (integer, default 0)
- `invoke` — ordered list of invocation strings
- `env` — optional table of extra environment variable assignments
- `mock` — ordered list of `[[scenario.mock]]` declarations

The `id` field is scoped to its suite directory.
The `Trial` name registered with `libtest-mimic` is `<suite>::<id>::<index>`,
where `<index>` is the 0-based position of the invocation in the `invoke` array —
e.g. `selectors::nth-zero-warned::0` and `selectors::nth-zero-warned::1`.
Each invocation is a fully independent `Trial` with its own temp directory,
its own mock state, and no shared state with other invocations of the same scenario.

### Trial Registration

Each scenario is registered as one `Trial`.
A failing `Trial` reports the combined stdout and stderr of `run-scenario.sh`
as its failure message, preserving all diagnostic output from the bash layer.

### Temp Directory Lifecycle

For each `Trial`, the Rust layer creates a temp directory before invoking the
bash layer and removes it unconditionally on completion, whether the Trial
passes or fails.
Since each Trial corresponds to exactly one invocation, there is no
`invocation-N/` subdirectory structure — the temp directory is the invocation's
own isolated workspace.

The temp directory layout:

```
<tmpdir>/
  scenario.sh              # generated by Rust, sourced by run-scenario.sh
  bin/                     # generated mock scripts, prepended to PATH
  mock-state/              # call counter state files, one per mock command name
  input                    # copy of the input file for this invocation
  stdout                   # captured stdout
  stderr                   # captured stderr
  output                   # captured --output / tee target
```

### `scenario.sh` Generation

The Rust layer writes `scenario.sh` into the temp directory before invoking
`run-scenario.sh`.
`run-scenario.sh` sources `scenario.sh` at startup to obtain all scenario data.

`scenario.sh` MUST export the following variables:

**Scalar fields**

```bash
SUITE_DIR="/absolute/path/to/tests/<suite>"
SCRIPT="$SUITE_DIR/scripts/<script-file>"
INPUT_SRC="$SUITE_DIR/inputs/<input-file>"
STDOUT_GOLDEN="$SUITE_DIR/goldens/stdout/<stdout-value>"
STDERR_GOLDEN="$SUITE_DIR/goldens/stderr/<stderr-value>"
OUTPUT_GOLDEN="$SUITE_DIR/goldens/output/<output-value>"
EXPECTED_EXIT_CODE=0
```

`<stdout-value>`, `<stderr-value>`, and `<output-value>` are the raw values from
the manifest — they may contain a glob suffix (e.g. `uuid-v5-line.*`).
The bash layer resolves them against the golden directories at comparison time.

**Invocation**

```bash
INVOCATION='qed "$(cat "$SCRIPT")" < "$INPUT" 2> "$STDERR" | tee "$OUTPUT" > "$STDOUT"'
```

The value is a single-quoted string preserving `$SCRIPT`, `$INPUT`, `$STDOUT`,
`$STDERR`, and `$OUTPUT` as unexpanded references.
`run-scenario.sh` sets these variables in scope before `eval`-ing the invocation.

The Rust layer selects the correct invocation string by its 0-based index when
generating `scenario.sh` for each Trial.

**Mock declarations**

```bash
MOCK_COUNT=2

MOCK_0_COMMAND="upcase"
MOCK_0_INPUT="$SUITE_DIR/mocks/input/bar.txt"   # empty string if not declared
MOCK_0_STDOUT="$SUITE_DIR/mocks/stdout/bar-upper.txt"
MOCK_0_STDERR=""                                 # empty string → no stderr output
MOCK_0_EXIT_CODE=0
MOCK_0_EXPECTED_ARGS_COUNT=0                     # 0 means no argument validation

MOCK_1_COMMAND="sortfile"
MOCK_1_INPUT="$SUITE_DIR/mocks/input/three-lines.txt"
MOCK_1_STDOUT="$SUITE_DIR/mocks/stdout/three-lines-sorted.txt"
MOCK_1_STDERR=""
MOCK_1_EXIT_CODE=0
MOCK_1_EXPECTED_ARGS_COUNT=1
MOCK_1_EXPECTED_ARG_0='${QED_FILE}'    # single-quoted — expanded at mock validation time
```

`MOCK_N_EXPECTED_ARG_M` values that reference `${QED_FILE}` MUST be
single-quoted so the literal string `${QED_FILE}` is stored rather than
expanded at generation time.
The generated mock script expands `${QED_FILE}` at argument validation time,
when it is present in the mock's own environment (injected by qed).

**Extra environment variables**

```bash
export QED_PATTERN="bar"
```

Values from the manifest `env` table are expanded from the Rust process's
own environment at `scenario.sh` generation time.
The resulting `export` statements export fixed strings — no further expansion
occurs when `scenario.sh` is sourced.

### Execution and Reporting

The Rust layer invokes:

```
bash tests/harness/run-scenario.sh <tmpdir>
```

`run-scenario.sh` sources `<tmpdir>/scenario.sh` via the `<tmpdir>` argument.

If `run-scenario.sh` exits zero, the `Trial` passes.
If it exits non-zero, the `Trial` fails.
The combined stdout and stderr of `run-scenario.sh` are captured and reported
as the failure message.

## Bash Layer: `run-scenario.sh`

### Startup

`run-scenario.sh` receives the temp directory path as `$1` and sources
`$1/scenario.sh` at startup.
All paths written to `scenario.sh` are absolute, so `run-scenario.sh` does
not need to know the workspace root.

### File Setup

`run-scenario.sh` generates all mock scripts for this invocation.
See [Mock Scripts](#mock-scripts).

The input file is copied from `INPUT_SRC` into the temp directory:

```bash
cp "$INPUT_SRC" "<tmpdir>/input"
```

### Execution

**Variable injection**

Before `eval`-ing the invocation, `run-scenario.sh` sets:

```bash
INPUT="<tmpdir>/input"
STDOUT="<tmpdir>/stdout"
STDERR="<tmpdir>/stderr"
OUTPUT="<tmpdir>/output"
```

**Environment setup**

```bash
export MOCK_STATE_DIR="<tmpdir>/mock-state"
export PATH="<tmpdir>/bin:$PATH"
```

`MOCK_STATE_DIR` must be exported so mock subprocesses can locate the call
counter state files.
`PATH` is prepended with the mock `bin/` directory so mock scripts shadow
any real commands of the same name.

If `scenario.sh` contains `export` statements from the manifest `env` table,
these are already in the current shell's environment after sourcing and are
inherited by the invocation subprocess.

**Invocation execution**

```bash
eval "$INVOCATION"
ACTUAL_EXIT=$?
```

**Exit code assertion**

```bash
if [[ "$ACTUAL_EXIT" -ne "$EXPECTED_EXIT_CODE" ]]; then
  echo "FAIL [$SCENARIO_ID]: exit code $ACTUAL_EXIT, expected $EXPECTED_EXIT_CODE" >&2
  exit 1
fi
```

**Golden comparison**

After a passing exit code assertion, stdout, stderr, and output are each
compared against their golden.
See [Golden Comparison](#golden-comparison).

If any comparison fails, `run-scenario.sh` prints a diagnostic and exits 1.

### Post-Invocation Checks

After the invocation completes, `run-scenario.sh` checks for unconsumed mock
declarations.
See [Unconsumed Mock Reporting](#unconsumed-mock-reporting).

## Mock Scripts

### Generation

`generate-mock.sh` is called once per distinct command name appearing in
`MOCK_COUNT` declarations.
It generates a bash script at `<tmpdir>/bin/<command>` and makes it executable.

The generated script embeds all declarations for that command name directly
as bash variables, so it is self-contained and requires no external data file.

### Call Counter Mechanism

Each generated mock script maintains a call counter in a state file at
`$MOCK_STATE_DIR/<command>.count`.

On each invocation the mock:

1. Reads the current count from the state file (default 0 if absent)
2. Increments the count
3. Writes the new count back to the state file
4. Dispatches to the declaration at index `count - 1`
5. If `count` exceeds the number of declarations for this command,
   writes a diagnostic to stderr and exits non-zero

State file reads and writes MUST be atomic within the mock script
(using a temp file and `mv`) to avoid races when mocks run in parallel.

### Input Validation

If `MOCK_N_INPUT` is non-empty, the mock validates the content delivered to it.

The mock determines the delivery mechanism at runtime:

- If `$QED_FILE` is set in the mock's environment, input was delivered via file
  materialization — the mock reads from `$QED_FILE`
- Otherwise, input was delivered via stdin — the mock reads from stdin

In both cases the actual content is compared byte-for-byte against the file
at `MOCK_N_INPUT`.
If the comparison fails, the mock writes a diagnostic to stderr and exits non-zero.

### Argument Validation

If `MOCK_N_EXPECTED_ARGS_COUNT` is greater than zero, the mock validates its
positional arguments against the expected values.

For each argument index `M`, the mock evaluates `MOCK_N_EXPECTED_ARG_M` in the
mock's own environment at validation time.
This is significant for values that contain `${QED_FILE}`: the value is stored
as a literal string in the generated script (single-quoted in `scenario.sh`,
transferred as-is to the mock script), and `eval` or double-quote expansion
is used at validation time so `${QED_FILE}` resolves to the path qed injected
into the mock's environment, not to any value at generation time.

If actual and expected arguments do not match, the mock writes a diagnostic
to stderr and exits non-zero.

### Unconsumed Mock Reporting

After all invocations complete, `run-scenario.sh` checks whether every
declared mock call was consumed.

For each distinct command name, `run-scenario.sh` reads
`$MOCK_STATE_DIR/<command>.count` and compares it against the number of
declarations for that command.
If the count is less than the number of declarations, the undispatched
declarations are reported as a harness error and `run-scenario.sh` exits 1.

## Golden Comparison

Golden comparison is performed by `compare-golden.sh`, called once per
golden file after each invocation.
It receives the actual output file path and the golden file path (which may
contain a glob suffix).

### Exact Comparison (`.txt`)

If the resolved golden filename ends in `.txt`, the comparison is byte-for-byte:

```bash
diff "$actual" "$golden"
```

Any difference causes a failure with the diff output as the diagnostic.

### Pattern Comparison (`.pattern`)

If the resolved golden filename ends in `.pattern`, the golden file contains
a single anchored regex (RE2 syntax).
The comparison reads the actual output as a single string and tests whether
it matches the regex in full:

```bash
actual_content=$(cat "$actual")
pattern=$(cat "$golden")
if ! echo "$actual_content" | grep -qP "^(?:${pattern})$"; then
  echo "FAIL: actual output does not match pattern"
  ...
fi
```

The regex is implicitly anchored — the entire actual output must match, not
just a substring.

### Glob References (`.*`)

If the golden filename value in the manifest ends in `.*`
(e.g. `uuid-v5-line.*`), the harness resolves the glob against the golden
directory and runs every comparison it finds.
A `uuid-v5-line.txt` golden triggers an exact comparison; a
`uuid-v5-line.pattern` golden triggers a pattern comparison.
All matching comparisons must pass.
It is a harness error if the glob matches no files.

---

## Failure Reporting

All diagnostic output is written to stderr so it is captured by the Rust
layer and included in the `cargo test` failure message.

Each diagnostic line identifies the Trial by its full name:

```
FAIL [selectors::nth-zero-warned::0]: exit code 1, expected 0
FAIL [external-processors::external-file::0]: stdout does not match golden
--- expected (goldens/stdout/bar-upper.txt)
+++ actual
...
FAIL [external-processors::external-replace-pipeline::0]: unconsumed mock declaration: extract-name call 2 was never reached
```

`run-scenario.sh` exits 0 only when all invocations pass all assertions and
no unconsumed mock declarations remain.
Any failure causes an immediate exit 1.
